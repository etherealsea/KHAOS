# 2026-04-24：Iter15 smoke、horizon 链路排查与交接记录

## 1. 今日核心结论

今天的工作主线不是继续扩写模型结构，而是集中排查 `iter15_event_first` 在 `smoke` 阶段的工程链路与 `horizon` 失效问题。结论可以分成两部分：

1. **训练主链路已经能正常运行**。`smoke_v9` 是一轮完整有效的 `smoke`，2 个 epoch 跑完，评估、报告、manifest 和指标文件都完整落盘。
2. **`horizon` 仍未真正打通**。虽然已经定位并修复了 `runner` 层一个关键漏项，但当前最后一轮有效 `smoke` 仍然没有产出有效的 `horizon_profile` 汇总，说明问题已经从“入口没开”收缩为“元信息没有贯穿到最终 summary 落盘”。

这意味着：**现在工程上已经接近可正式训练，但科研目标上还不应仓促进入完整训练**。当前系统已经不再是完全错误的版本，但还不是“真正带 horizon-aware 语义”的稳定版本。

### 2.2 `reversion` 分支已摆脱早期塌缩

在更早的版本里，`reversion` 曾经出现主输出极度接近零、阈值畸形压到极小值的问题。当前 `v9` 的 smoke 结果说明，这一问题已经显著缓解：

- `reversion_precision` 与 `reversion_f1` 不再整体塌成近零
- `iter15` 的输出/损失/评分链至少已经回到“能学到事件概率”的状态

这部分说明前面的 `iter15` 输出语义与损失逻辑修复方向是有效的。

### 2.3 终端“像卡死”的主因已经识别清楚

此前多次出现“终端一直转但没有结果”的现象，核心原因不是训练循环本身必然卡死，而是：

1. 在同一个终端里复用查询命令会打断正在运行的训练命令。
2. PowerShell 的异步管道/`Tee-Object` 方式在部分轮次里会制造“无声退出”或“日志不完整”的假象。

因此，后续 smoke 或正式训练应遵守两条原则：

1. **训练终端专用，不再复用查询命令。**
2. **优先使用前台直跑或明确的独立日志监控，不再依赖可疑的异步管道。**

## 3. 今日最重要的新发现

### 3.1 `runner` 曾经根本没有把 `horizon_search_spec` 传进训练

这是今天最关键的突破点。

虽然 `scripts/run_iter14_multiasset_closed_loop.py` 的 `SMOKE_PRESET` / `FORMAL_PRESET` 里都已经写了：

- `horizon_search_spec = "3,5,8,13,20"`
- `horizon_family_mode = "adaptive_resonance"`

但实际排查发现：

1. `resolve_effective_config()` **没有把** `horizon_search_spec` 合入最终 `config`
2. CLI parser 里 **没有** `--horizon_search_spec`

这导致此前所有 smoke 虽然表面上在跑 `split_by_month + iter15`，但训练阶段实际上始终拿不到 `global_horizon_grid`。

这个问题已经在今天修复到 `runner` 层，后续训练命令里已经能明确看到：

- `--horizon_search_spec 3,5,8,13,20`
- `--horizon_family_mode adaptive_resonance`

并且 `train.py` 已经真实打印出：

- `Global horizon grid size: 5`

这说明：**`horizon` 的入口配置链终于第一次被真正打通了。**

### 3.2 `split_by_month` 路径的 horizon 问题，不再是“入口没开”，而是“落盘链仍断”

今天之前，`horizon` 失效很容易被误判成模型、损失或阈值问题。但排查后发现，真正的问题是分层的：

1. 早期是 `split_by_month` 路径没有把 horizon-aware 逻辑真正接入主流程。
2. 随后又确认 `runner` 根本没把 `horizon_search_spec` 传进训练。
3. 现在前两层已经修到了位，问题收缩成：
   - 训练已经识别 `Global horizon grid size: 5`
   - 但最终 `horizon_discovery_summary.json` 仍然没有形成有效记录

所以当前最值得接手继续查的，不再是外围配置，而是：

- `dataset_meta -> horizon_registry -> summary artifact` 这条元信息传递链

## 4. 当前仍未解决的问题

### 4.1 `smoke_v9` 的 `horizon` 仍然是失效状态

`v9` 虽然是完整有效的 smoke，但它的 `horizon_discovery_summary.json` 仍然是：

- `status = inactive`
- `global_horizon_grid = null`
- `records = {}`

并且两个 epoch 里的：

- `horizon_entropy_breakout_mean = 0.0`
- `horizon_entropy_reversion_mean = 0.0`

这说明 `v9` 不能作为“真正 horizon-aware 版本已经成立”的证据，它只能证明训练主链路已经正常。

### 4.2 `smoke_v10` 只验证了入口修复，但没有形成最终产物

`v10` 的价值在于验证新修复是否真的进入训练。

目前已确认：

1. 它的训练命令里已经带上 `--horizon_search_spec`
2. 训练日志里已经打印 `Global horizon grid size: 5`

但当前仍没有看到 `v10` 的最终结果文件落盘，因此它还不能作为新的有效 smoke 基线。

这意味着：`v10` 更像是一轮“入口修复验证”，还不是可以直接拿来决定是否正式训练的最终 smoke 证据。

### 4.3 结构评分不再全是伪常数，但仍需继续校准

为了避免 `iter15` 继续借用 `iter14` 的伪结构项，今天已经把一部分结构评分从“全 1.0 的假安全感”里拉出来。

当前结果里已经能看到：

- `directional_floor_mean = 0.0`
- `directional_floor_quality = 0.25`
- `structural_guard_quality = 0.55`

这比过去的“全部恒等于 1.0”更真实，但也说明这套结构评分还在早期，需要进一步校准，不能马上把它当成正式长训的成熟选模标准。

## 5. 对神经网络当前状态的判断

### 5.1 可以确认有效的部分

当前 `iter15_event_first` 已经不再是旧版“方向收益回归”那种明显错位的训练目标。它至少已经实现了：

1. 事件优先的主表达方式
2. `reversion` 输出不再系统性塌缩
3. 训练/评估/阈值拟合/报告生成这条工程主链基本跑通

### 5.2 还不足以支持正式完整训练的部分

当前最关键的短板不是 loss 数值本身，而是：

1. `horizon` 仍未真正成为有效监督与有效 summary
2. 结构评分体系仍处在校准中
3. 最后一轮验证 `horizon` 入口修复的 `v10` 还没有形成可交付的 smoke 结果

因此现在如果直接开始正式完整训练，本质上更像是在放大一个“主链路已通但 horizon 目标仍未真正激活”的版本。

### 8.4 今日新增的稳定结论

截至这次补充验证，可以把问题判断进一步收缩为：
