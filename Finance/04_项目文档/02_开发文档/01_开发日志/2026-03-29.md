# 2026-03-29

## 跨平台载体调研进入“约束重排”阶段

### 1. 当前工作主线已经从 Pine 映射问题延伸到平台承载问题

- 本轮推进的重点，已经不再只是继续修补 `KHAOS.pine` 的表达失真，而是开始系统审视：后续到底应该把 KHAOS 放在哪一类平台上承载，才能尽量不牺牲训练语义、性能和跨市场扩展性。
- 这意味着当前讨论的重心，已经从“Pine 怎么画得更像模型”进一步转向“是否应该继续把 Pine 视为核心落地端，以及有哪些外部平台值得纳入候选”。
- 从研发路径上看，这是一次任务层级上移：不再只优化单一脚本，而是在重新定义未来部署载体。

### 2. 当前阶段重新明确的平台筛选约束

- 本轮没有继续提前替项目做平台选择，而是先把真正有效的筛选条件重新写清楚。
- 当前约束已经收敛为以下几条：
  - 尽可能免费或低付费；
  - 数据最好免费，且覆盖面尽量广、尽量实时；
  - 国内可使用，允许挂梯子，但不希望有过于复杂的注册、开户或接入障碍；
  - 最好支持 `Python`、`Rust` 这类高性能或高研发效率语言接入；
  - 不只是能算信号，还要能把指标结果稳定显示在图表端。
- 这个约束集非常重要，因为它和此前只从“专业能力最强”出发去看平台完全不同。当前更关注的是“低门槛可落地”而不是“理论上最完美”。

## 本轮调研形成的阶段性认识

### 1. 目前没有单一平台能完全同时满足全部条件

- 这是当前最关键的现实判断。
- 想同时拿到“低成本、免费广覆盖实时数据、国内易用、原生 Python/Rust 编写、直接图表显示”这五项，公开市场上的单个平台基本都做不到。
- 因此后续必须接受“以主约束优先级做取舍”，而不是继续假设存在一个天然完美平台。

### 2. FastBull 更像内容与白标能力候选，不像核心脚本运行平台

- 目前查到 FastBull 有白标、iFrame 插件、行情、经济日历、内容与 AI signals 等能力。
- 但从公开信息看，它更像“资讯 / 行情 / 内容门户 + 白标输出”平台，而不是成熟的自定义指标脚本生态。
- 因此它更适合被理解为外部展示层或门户层候选，而不适合作为 KHAOS 真值逻辑的主要运行端。

### 3. cTrader、MT5、Quantower 属于真正值得继续细查的终端型候选

- `cTrader` 当前最值得关注的点，不是它已经被选定，而是它同时满足了几个关键条件：平台门槛相对可控、实时数据通常由 broker 提供、并且官方公开支持 `Python` 编写 algo 与 indicator。
- `MetaTrader 5` 的优势在于普及度、终端免费使用和 broker 覆盖广，但其图表指标主表达语言仍然是 `MQL5`，并不等价于直接用 Python 写图上指标。
- `Quantower` 的优势在于多连接、多数据源和专业终端能力，但完整功能和第三方市场数据并不天然免费，因此更像“能力强但成本未必低”的路线。

### 4. B2B 图表与数据基础设施仍是长期路线，但不是当前低门槛答案

- `DXcharts`、`Barchart`、`dxFeed` 这类方案在长期产品化上依然有价值，因为它们更接近“自有前端 + 自有能力组合”的底层路线。
- 但这条路更偏向搭建自己的平台，而不是直接找到一个现成低成本终端来承载 KHAOS。
- 因此它们更适合作为长期演化方向，而不是当前阶段的低门槛首选答案。

## 当前最新任务标记

### 1. 任务状态

- 当前任务还停留在“约束与候选平台收缩”阶段，**尚未进入最终平台拍板阶段**。
- 当前不应把任何单一平台提前写成最终路线，避免在数据成本、国内可用性和开发语言支持这些约束尚未核实完之前过早收敛。

### 2. 当前最新任务

- **任务一：继续做候选平台能力对照表。**
  - 重点对象至少包括：`cTrader`、`MetaTrader 5`、`Quantower`、`FastBull`。
  - 如有必要，再补充 `TrendSpider`、`NinjaTrader` 作为扩展参照。

- **任务二：把“能不能显示指标”与“能不能用 Python / Rust 写”拆开核查。**
  - 很多平台支持外部 API，不等于支持把指标原生显示在图表里。
  - 也有一些平台支持 Python 做外部分析，但图表指标层仍然要回到平台自有语言。

- **任务三：重点核实真实使用门槛，而不是只看官网功能页。**
  - 后续需要继续确认国内是否可注册、是否必须开户、是否强依赖特定 broker、免费数据是否真的实时、免费层是否足够支撑研究验证。

- **任务四：为后续部署路线保留双分支判断。**
  - 分支 A：找一个低门槛现成终端，先把 KHAOS 指标低成本落地。
  - 分支 B：如果现成终端始终无法同时满足数据、语言和显示要求，则重新回到“自建真值引擎 + 外部终端显示层”的架构路线。

## 当前阶段判断

- 截至今天，平台问题已经不能再只看“哪个平台看起来像 TradingView”，而必须回到“哪个平台最接近当前真实约束”。
- 当前最重要的不是立即选平台，而是继续把“免费性、数据、国内可用性、语言支持、图表显示能力”这五项拆开核实。
- 因此，下一轮工作的正确入口，不是继续抽象讨论平台优劣，而是把候选平台做成可比、可删减、可收敛的能力清单。
