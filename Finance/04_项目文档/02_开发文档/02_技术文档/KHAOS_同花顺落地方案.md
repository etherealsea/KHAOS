# KHAOS 同花顺使用与落地方案

## 1. 文档目的

本文面向当前 KHAOS / PI-KAN 项目，回答一个更具体的问题：

**在暂时拿不到“可直接用 Python 编写图表指标”的合适平台时，如何把 KHAOS 有效落地到同花顺环境中使用？**

本文不再讨论“同花顺能否无损承载完整模型”，而是直接给出：

- 同花顺里适合承担什么职责；
- 不适合承担什么职责；
- 当前阶段最可执行的落地路线；
- 短期、中期的实施顺序。

---

## 2. 先给结论

### 2.1 推荐定位

当前最合理的定位不是：

- 把同花顺当成 Python 真值引擎的直接承载平台；

而是：

- **把同花顺当作国内终端展示、预警、选股和人工读图入口；**
- **把 Python 继续保留为研究、校准、参数导出和真值对照层。**

### 2.2 推荐路线

建议采用三层方案：

1. **同花顺公式层**
   - 承担图表显示、状态提示、预警条件、选股条件；
   - 实现 KHAOS 的轻量终端表达版。

2. **Python 真值层**
   - 承担特征计算、参数校准、回测验证、版本管理；
   - 继续作为 KHAOS 的权威语义来源。

3. **参数同步层**
   - 将 Python 侧确认过的阈值、状态映射、周期说明同步到同花顺公式版；
   - 控制同花顺公式与主模型之间的语义漂移。

一句话总结：

**同花顺负责“看、筛、提醒”，Python 负责“算、校、对齐”。**

---

## 3. 为什么当前不应期待“同花顺里直接用 Python 写指标”

### 3.1 从项目本身看

当前项目已经明确：

- 训练主链路在 Python；
- physics 特征是 14 维统一特征；
- 模型输出是 breakout / reversion 双核；
- Pine 端都已经被定义为“终端解释器”，不是训练本体。

因此，同花顺也不应被要求承担：

- 完整神经网络推理；
- 完整物理特征真值定义；
- 多版本模型治理；
- 跨终端一致性标准源。

### 3.2 从同花顺能力边界看

基于当前可公开确认的信息，可以做出如下判断：

- SuperMind 官方页面明确支持 **Python 策略回测、研究环境和本地 SDK**；
- iFinD 数据接口官方文档明确支持 **Python 等多语言数据接口**；
- 同花顺指标平台公开展示的仍然是 **公式源码形态的指标**。

由此可以合理推断：

- **同花顺的 Python 能力主要面向投研、回测、数据接口与策略开发；**
- **标准终端图表指标生态仍然主要是公式语言，而不是“Python 直接写图上指标”。**

这不是说同花顺不能接入 Python，而是说：

**Python 更像研究与策略环境，同花顺终端更像公式与展示环境。**

---

## 4. 当前项目落地到同花顺时最重要的现实约束

### 4.1 约束一：当前训练资产并不以 A 股为核心

现有训练主线优先覆盖的资产更接近：

- BTC / ETH
- GOLD / XAU
- OIL / WTI
- SPX / NDX / DJI / ES

这意味着：

- 如果直接把当前阈值原样搬到同花顺 A 股环境，
- 很可能会遇到分布漂移、波动结构不同、交易时段不同的问题。

因此：

**同花顺落地不能只做公式移植，还必须做一轮“国内市场口径校准”。**

### 4.2 约束二：同花顺更适合做状态表达，不适合做完整真值重建

当前 KHAOS 的最佳表达方式，不是把每一层 physics 细节都塞进公式，而是保留：

- 双核状态；
- 中性过滤；
- 方向确认；
- 短确认锚点；
- 必要的能量与动量读图入口。

### 4.3 约束三：周期体系需要贴近国内终端习惯

当前项目主周期为：

- 5m
- 15m
- 1h
- 4h
- 1d

同花顺侧建议映射为：

- 5 分钟
- 15 分钟
- 60 分钟
- 240 分钟
- 日线

其中：

- 240 分钟可作为 4h 的近似终端对应；
- 但国内市场非 24/7 连续交易，必须接受其与原训练市场结构并不完全同构。

---

## 5. 推荐的同花顺使用方案

## 5.1 方案 A：公式指标版

这是**当前最应该优先落地**的方案。

### 目标

在同花顺终端中实现一个 **KHAOS_THS** 公式指标，用于：

- 看图；
- 判读当前状态；
- 提醒 breakout / reversion 切换；
- 服务人工交易决策。

### 建议功能

公式版不追求还原完整模型，只保留以下核心输出：

- `Breakout`
  - 方向无关；
  - 强调压缩后释放；
  - 作为动能环境预警。

- `Reversion Bull`
  - 极端失衡后的看多回归环境；
  - 属于确认型回归，不做任意抄底。

- `Reversion Bear`
  - 极端失衡后的看空回归环境；
  - 属于确认型回归，不做任意摸顶。

- `Neutral / Filter`
  - 没有足够把握时明确输出中性；
  - 不再把“图上没东西”误读成没有信息。

### 图表建议

建议做成 **副图主指标**，而不是一开始就做复杂主图。

推荐视觉结构：

1. 一条主动能线
   - 对应当前状态驱动的主动量基线。

2. 一组能量柱
   - 用于表达 breakout impulse 或 reversion setup 的强弱变化。

3. 状态着色
   - 橙色：Breakout
   - 蓝色：Bull Reversion
   - 紫色：Bear Reversion
   - 灰色：Neutral

4. 图标或文字标记
   - `BK`
   - `RB`
   - `RS`
   - `NF`

### 公式层实现原则

- 保持当前 Pine 的双核状态机，不回退到旧版 Force 叙事；
- 保留 20 周期慢背景；
- 保留 2~3 周期锚点确认；
- breakout 继续保持“无方向释放核”；
- reversion 才做 bull / bear 方向结算；
- neutral/filter 必须是显式状态。

### 公式层允许的近似

可以接受的近似包括：

- Hurst 使用终端可计算代理；
- MLE、Entropy 继续使用在线代理式近似；
- EKF 采用简化残差代理或递推近似；
- 背景色不足时改用带状柱、色条、图标提示。

不应做的事情包括：

- 为了“更像模型”把公式堆到难以维护；
- 为了好看人为增强高周期 breakout；
- 让同花顺公式反向定义主系统语义。

## 5.2 方案 B：同花顺预警 / 选股版

这是公式指标版之外，**非常适合同时落地**的一层。

### 目标

把 KHAOS 压缩成可执行的终端筛选规则，用于：

- 盘中预警；
- 条件选股；
- 自选池轮询；
- 板块内信号扫描。

### 建议输出 4 类条件

1. `Breakout 启动`
   - 当前进入 breakout，且上一根不在 breakout。

2. `Bull Reversion 确认`
   - 当前进入 bull reversion，且方向差值充分明确。

3. `Bear Reversion 确认`
   - 当前进入 bear reversion，且方向差值充分明确。

4. `Neutral 过滤`
   - 当前没有有效主导状态，不纳入激进筛选。

### 价值

这层的价值很高，因为它能直接解决“同花顺终端如何被真正用起来”的问题：

- 图上指标负责解释；
- 预警和选股负责把信号找出来。

对于国内终端而言，这往往比单纯把图画出来更实用。

## 5.3 方案 C：SuperMind / Python 研究辅助版

这是**条件式可选方案**，不建议把它当作当前主落地依赖。

### 适用定位

如果你后续能够拿到 SuperMind 可用权限，或者有 iFinD 数据接口条件，那么它很适合承担：

- A 股 / 期货数据研究；
- Python 回测；
- 参数校准；
- 信号导出；
- 选股池生成；
- 真值对照。

### 不建议当前把它当主方案的原因

从官方帮助信息看：

- SuperMind 本地 SDK 有明显申请门槛；
- 更多面向机构化策略研发与回测；
- 当前是否适合你立即直接使用，需要单独核实权限与可用性。

因此当前更稳妥的判断是：

- **把 SuperMind 当作 Python 研究增强层；**
- **不要把它当作“桌面同花顺图上指标直接替代品”。**

## 6. 对 KHAOS 的具体落地形态建议

建议不要只做一个“大而全公式”，而是拆成三份。

### 6.1 `KHAOS_THS_CORE`

职责：

- 副图主指标；
- 输出双核状态和能量；
- 服务人工读图。

核心元素：

- 主动能线；
- 能量柱；
- breakout / reversion / neutral 状态色；
- 必要图标标记。

### 6.2 `KHAOS_THS_ALERT`

职责：

- 预警条件公式；
- 只回答“当前是否值得提醒”。

建议条件：

- breakout 新启动；
- bull reversion 新确认；
- bear reversion 新确认。

### 6.3 `KHAOS_THS_SELECT`

职责：

- 选股 / 扫描条件公式；
- 服务批量筛选而不是精细可视化。

建议条件：

- 状态明确；
- 中性过滤关闭；
- 最近若干周期内存在锚点确认；
- 剔除明显噪声触发。

---

## 7. 推荐实施顺序

## 第一步：先做同花顺副图 MVP

目标：

- 先把 KHAOS 的状态语言搬进去；
- 不追求完整模型；
- 只验证“能否看得懂、能否稳定跑、能否不扭曲主语义”。

MVP 必须包含：

- breakout / reversion / neutral 三态；
- bull / bear 方向区分；
- 20 周期主背景；
- 2~3 周期短确认；
- 至少一条主动能线和一组能量柱。

## 第二步：做预警与选股

目标：

- 把指标从“能看”推进到“能用”。

优先增加：

- 突破启动预警；
- 回归确认预警；
- 条件选股公式。

## 第三步：做国内市场口径校准

这一阶段非常关键。

建议最少分两类校准：

1. **市场校准**
   - A 股
   - 股指期货 / 商品期货

2. **周期校准**
   - 5m / 15m
   - 60m / 240m
   - 日线

重点核查：

- breakout 是否在高周期被压到几乎不触发；
- reversion 是否在 A 股里过于频繁；
- neutral/filter 是否足够诚实；
- 国内停盘与跳空是否显著影响代理特征。

## 第四步：再决定是否接 SuperMind / iFinD

只有当下列目标成立时，再去接 Python 增强层最划算：

- 同花顺公式版已经证明“读图有价值”；
- 同花顺预警版已经证明“筛选有价值”；
- 你确实需要更系统的参数管理、选股输出或回测支持。

## 8. 推荐的工程分工

### 8.1 同花顺侧负责

- 图表展示；
- 状态解释；
- 预警触发；
- 条件选股；
- 面向国内终端用户的可读表达。

### 8.2 Python 侧负责

- 特征真值；
- 阈值校准；
- 版本管理；
- 分市场参数包；
- 回测验证；
- 同花顺公式结果对照。

### 8.3 二者之间的同步内容

建议固定同步以下对象：

- `feature_version`
- `model_version`
- 周期映射说明
- breakout / reversion 阈值
- neutral gap
- anchor_len
- 各市场备注

---

## 9. 风险与取舍

### 风险一：A 股与原训练市场分布不一致

这是当前最大的真实风险。

如果不校准：

- 同花顺版容易出现阈值失真；
- breakout 可能过稀或过密；
- reversion 可能偏多。

### 风险二：同花顺公式表达能力有限

这会导致：

- EKF 等复杂过程只能近似；
- 复杂上下文注意力无法原样保留；
- 视觉表达要比 Pine 更保守。

### 风险三：如果过度追求“公式还原模型”，维护成本会迅速失控

因此推荐原则应始终是：

- **保留主语义；**
- **接受代理实现；**
- **通过 Python 做校准，而不是通过公式做真值定义。**

---

## 10. 当前最推荐的最终方案

如果目标是尽快让 KHAOS 在同花顺里可用，那么当前最推荐的不是等待“同花顺直接 Python 指标能力”，而是：

### 短期方案

- 做一套 `KHAOS_THS_CORE + ALERT + SELECT` 三件套；
- 同花顺作为终端展示与筛选层；
- 先在国内终端把使用路径跑通。

### 中期方案

- 用 Python 补上国内市场校准；
- 输出分市场、分周期参数；
- 作为同花顺公式版的校准母体。

### 长期方案

- 如果后续拿到合适的 SuperMind / iFinD / 机构接口条件，
- 再把 Python 研究与信号导出层系统接上；
- 形成“Python 真值层 + 同花顺终端层”的稳定双轨。

一句话结论：

**同花顺现在最适合做 KHAOS 的国内终端消费层，而不是 Python 真值引擎替身。先把它做成好用的图表、预警、选股入口，再用 Python 做校准和增强，是当前最稳、最快、最现实的路线。**
