# KHAOS A股训练数据选取与构建报告

更新日期：`2026-04-07`

## 1. 文档目的

本报告用于回答一个在 GitHub-only 协作场景里非常关键的问题：

**在不直接上传训练数据的前提下，如何让后续协作者准确理解当前 A 股训练数据是如何被选取、构建与使用的。**

它的目标不是替代真实数据本身，而是明确以下内容：

- 当前 A 股训练数据的选取原则
- 主标的池与备选池的固定口径
- 周期、切分、清洗与覆盖校验规则
- 设计方案与当前活跃 runner 之间的差异
- 在没有原始数据的情况下，后续协作者应如何保持口径一致

---

## 2. 为什么必须单独定义 A 股训练数据

KHAOS / PI-KAN 项目早期与 `iter9` 阶段的训练覆盖过多类资产，包括：

- `BTC / ETH`
- `XAU / GOLD`
- `WTI / OIL`
- `SPX / NDX / DJI / ES`

但当前工程已经明确转向 **A 股专用 teacher 网络**。原因不是市场偏好，而是数据分布本身发生了根本变化：

- A 股不是 24/7 连续交易市场
- 存在午休断档
- 存在涨跌停与一字板
- 竞价、公告、复牌跳空对短周期结构影响更大
- “假突破”与“极端失衡后的反抽/回补”在 A 股更常见

因此，如果训练数据口径不单独定义，后续任何只阅读 GitHub 的协作者都很容易误把当前模型理解成“通用多资产 teacher 的小修小补”，这会直接造成：

- 对标签含义的误读
- 对周期表现的误读
- 对同花顺蒸馏可行性的误读

## 4. 选取原则与排除规则

### 4.1 选取原则

当前 A 股训练池的基本原则是：

- 流动性足够高
- 历史长度尽量长
- 行业结构足够分散
- 尽量避免单一行业主导 teacher 语义
- 优先选择沪深两市核心大盘股与行业龙头

### 4.2 默认排除对象

以下类型默认不作为第一版正式训练主标的：

- `ST` 与退市整理个股
- 长期停牌或数据长期缺失标的
- 上市历史过短、难以形成完整时间切分的标的
- 极端题材化、流动性失真严重的样本

### 4.3 特殊样本的处理态度

以下现象不是简单“删掉”，而是按训练语义区别处理：

- 一字涨停 / 一字跌停：
  可保留为状态参考，但不作为 breakout 正样本主来源
- 长停牌 / 复牌大跳空：
  需要降权或屏蔽其对可执行标签的主导
- 低成交量死盘阶段：
  不应轻易标为有效 breakout
- 午休断档：
  不能被误识别成真实回归确认

## 5. 周期体系：设计口径与当前执行口径

当前仓库中与 A 股训练数据相关的周期存在两个层次。

### 5.1 设计口径

在 A 股专用训练方案文档里，第一版主训练周期被定义为：

- `15m`
- `60m`
- `1d`

并明确提出：

- `5m` 作为第二版补充周期
- `240m` 暂不作为第一版核心训练周期

这个设计意图是先确保主结构在较稳周期上学会相变语义，再逐步引入更高噪声的短周期。

### 5.2 当前活跃执行口径

随着 `iterA5` 和 `teacher-first` 系列实验推进，当前活跃 runner 已经使用：

- `5m`
- `15m`
- `60m`
- `240m`
- `1d`

对应实现可见于：

- [`Finance/03_实验与验证/脚本/测试与临时脚本/run_iterA5_ashare_train.py`](../../../03_实验与验证/脚本/测试与临时脚本/run_iterA5_ashare_train.py)
- [`Finance/03_实验与验证/脚本/测试与临时脚本/run_teacher_first_ashare_ablation_train.py`](../../../03_实验与验证/脚本/测试与临时脚本/run_teacher_first_ashare_ablation_train.py)

### 5.3 如何理解这两层差异

后续协作者应这样理解：

- **方案文档中的 `15m / 60m / 1d`** 代表最初的稳态设计边界
- **当前 runner 中的 `5m / 15m / 60m / 240m / 1d`** 代表已经进入实验验证阶段的实际工程口径

因此，在没有数据文件时，如果你要对齐“当前主线训练数据”，应优先对齐当前活跃 runner，而不是只停留在最早的第一版设计口径。

## 6. 时间切分口径

当前 A 股训练集使用严格的时间顺序切分：

- Train：`<= 2023-12-31`
- Val：`2024-01-01 ~ 2024-12-31`
- Test：`>= 2025-01-01`

对应实现见：

- [`Finance/02_核心代码/源代码/khaos/数据处理/ashare_support.py`](../../../02_核心代码/源代码/khaos/数据处理/ashare_support.py)

这个切分口径是当前仓库里最重要的对齐边界之一。后续协作者即便没有原始 CSV，也不应擅自修改为随机切分或交叉打乱切分，否则：

- 时间外推能力会被高估
- A 股公告驱动与制度性断档的影响会被掩盖
- 同花顺蒸馏母体的实际泛化质量会被误判

## 7. 数据来源优先级

当前项目对 A 股数据来源采用明确的优先级：

1. 用户已有本地 A 股 CSV / 同花顺导出数据
2. 可用的 iFinD / 同花顺数据接口
3. BaoStock 等公开可获取数据

在当前实现里：

- `fetch_ashare_data.py` 会先检查本地导入目录
- 若未检测到 iFinD SDK 且本地 CSV 为空，则回退到公开数据抓取

相关入口：

- [`Finance/03_实验与验证/脚本/setup/fetch_ashare_data.py`](../../../03_实验与验证/脚本/setup/fetch_ashare_data.py)

需要特别说明的是：

- 数据来源可以更换
- 但最终落入 `training_ready` 的文件格式与字段口径必须统一

换句话说，**项目允许更换取数工具，不允许静默改变训练输入语义。**

## 8. 文件格式与 `training_ready` 口径

### 8.1 最终训练目录

当前活跃 A 股流程预期训练文件位于：

- `Finance/01_数据中心/03_研究数据/research_processed/training_ready/ashare`

### 8.2 文件命名

当前通过后缀识别周期，固定映射为：

- `5m -> *_5m.csv`
- `15m -> *_15m.csv`
- `60m -> *_1h.csv`
- `240m -> *_4h.csv`
- `1d -> *_1d.csv`

这意味着 `60m` 与 `240m` 在磁盘层分别使用 `1h` 和 `4h` 后缀，但在训练逻辑中会被统一识别为 `60m` 与 `240m`。

### 8.3 标准字段

必需字段：

- `time`
- `open`
- `high`
- `low`
- `close`
- `volume`

可选字段：

- `amount`
- `turnover`
- `adj_factor`

### 8.4 关于 `240m`

当前实现中的 A 股 `240m` 并不是简单按自然时钟切成两个 4 小时桶，而是按 **一个完整交易日的 240 个交易分钟** 聚合。这是为了避免午休断档把 A 股日内结构错误切裂。

## 9. 覆盖校验与备选补位逻辑

当前训练前不会直接假设“标的池齐全”，而是先做覆盖校验。

### 9.1 覆盖校验检查什么

覆盖报告会逐个检查：

- 文件是否存在
- 时间范围是否覆盖 train / val / test
- 每个标的是否在目标周期上完整可用

### 9.2 备选补位规则

若主标的池中出现覆盖不足：

- 会先尝试抓取缺失标的
- 仍不足时再从 6 只备选标的中补位

但补位的目标不是无限扩池，而是尽可能维持 **24 只正式可训练标的** 的完整性。

### 9.3 不允许的做法

当前项目明确不接受以下做法：

- 主标的池残缺但继续静默正式训练
- 用缺失周期的样本直接替代完整样本
- 因为取数不方便就把正式训练降级成极小样本训练

这也是为什么当前 runner 会在覆盖不足时直接报错，而不是悄悄继续。

## 10. 当前 sample weighting 如何反映数据选择

除了“选哪些标的、哪些周期”，当前项目还通过样本权重把 A 股微结构显式写入训练集。

### 10.1 `iterA5`

在 `iterA5` profile 中：

- `60m` 权重高于 `15m`
- `5m` 存在但带有更强边界惩罚
- 开收盘附近 `5m` bar 会被下调权重

### 10.2 `shortT_v1`

在 `shortT_v1` 中：

- 明显加强 `5m / 15m / 60m`
- 弱化 `240m / 1d`
- 更偏向短周期局部纹理与 breakout 相关样本

### 10.3 `shortT_balanced_v1 / v2`

在当前更推荐的 balanced 路线中：

- `60m` 被继续强调
- `15m` 保持重要位置
- `5m` 不再直接主导主 checkpoint 选择
- `240m / 1d` 仍保留为背景稳定层

这说明当前“训练数据选择”不是只有资产列表那么简单，而是一个由：

- 标的池
- 周期集合
- 边界过滤
- 事件权重
- 周期权重

共同组成的训练语义系统。

## 11. 当前最应被视为“对齐真值”的数据口径

如果后续协作者没有拿到真实训练数据，只能阅读仓库，那么当前最应被视为对齐真值的是：

### 固定不变量

- 市场限定为 A 股
- 主标的 24 只，备选 6 只
- 时间切分固定为 train/val/test 三段顺序切分
- 训练文件最终进入 `training_ready/ashare`
- 不允许混入 BTC / ETH / XAU / WTI / SPX / ES 等非 A 股样本

### 当前活跃执行口径

- 当前活跃 runner 使用 `5m / 15m / 60m / 240m / 1d`
- 原始抓取常见为 `5m / 15m / 60m / 1d`
- `240m` 通过 A 股会话聚合得到
- 覆盖报告决定是否允许正式训练

### 当前最重要的认知边界

没有原始数据时，不要臆造新的资产池、随机替换周期、或重写切分规则。应把本报告与以下文件一起视为当前训练数据的正式说明：

- [`A股训练标的清单.md`](A股训练标的清单.md)
- [`PIKAN_A股专用网络训练方案.md`](PIKAN_A股专用网络训练方案.md)

## 12. 给后续协作者的实用结论

如果你只有 GitHub 仓库，没有原始训练数据，那么你应默认：

1. 当前项目的正式 teacher 数据母体是 **纯 A 股**。
2. 当前固定资产口径是 **24 主标的 + 6 备选**。
3. 当前最活跃工程口径已经扩展到 **5m / 15m / 60m / 240m / 1d**。
4. 任何新实验若声称“可与当前主线对齐”，至少必须说明：
   - 是否沿用同一标的池
   - 是否沿用同一时间切分
   - 是否沿用同一 `training_ready` 命名规范
   - 是否沿用同一覆盖校验逻辑
5. 如果这些前提没有被满足，那么该实验最多只能算“旁路研究”，不能直接与当前主线 teacher 结果对齐比较。

## 13. 一句话总结

**当前仓库虽然不上传训练数据本身，但已经把 A 股训练数据的资产池、周期体系、切分边界、覆盖校验和样本权重逻辑固定下来了。后续协作者若要在没有原始数据的情况下与主线保持一致，应以这套口径为准，而不是重新发明一套“看起来差不多”的训练集。**
