# 跨平台指标引擎架构方案

## 1. 文档目的

本文档面向当前 KHAOS / PI-KAN 体系，回答一个核心工程问题：

**既然训练与解释能力主要在 Python 侧，而最终又希望落地到 Pine、同花顺等终端，什么样的架构既能保留研究能力，又能真正用于跨平台输出？**

本文给出的结论不是“把所有能力硬塞进单一平台”，而是采用一套**双轨混合方案**：

- **主轨：高保真真值引擎轨**
  - 由 Python 起步，后续可增加 Rust 运行时
  - 负责完整物理特征、双核推理、版本管理、参数生成、回测对齐
- **副轨：终端表达与交互轨**
  - 由 Pine / 同花顺等图表平台承接
  - 负责可视化、轻量告警、用户交互、终端侧阅读体验

一句话概括：

**复杂推理留在真值引擎，终端平台只负责“看得懂、跑得稳、能接收结果”。**

---

## 2. 现状与约束

### 2.1 当前系统已经具备的能力

从现有训练、physics、symbolic_extractor 与 `KHAOS.pine` 可以看到，系统已经形成一条较清晰的主链路：

- Python 训练端以 **14 维 physics 特征**为统一输入，并输出 **breakout / reversion 双核**结果
- 训练数据明确覆盖 **5m、15m、1h、4h、1d** 五类周期
- 训练与解释逻辑已经形成“**20 周期物理背景 + 2~3 周期短时确认**”的稳定工程共识
- Pine 端已经不是旧版单一 Force 叙事，而是转为 **双核状态表达**

### 2.2 代码层面反映出的关键约束

#### 约束一：训练端是多周期、多特征、双输出体系

- `train.py` 会从 `training_ready` 中筛选五类目标周期文件，并以 `output_dim=2` 训练 breakout / reversion 双核
- 训练时还会综合使用相关性、事件均值、hard negative 差值、F1 与 composite score 选模
- 这说明当前模型并不是“几个阈值拼出来的脚本”，而是一个**带选模逻辑的多目标训练系统**

#### 约束二：physics 特征并不只有几个简单指标

当前 physics 层至少包含以下 14 个统一特征：

- Hurst
- Volatility
- EKF_Vel
- EKF_Res
- Entropy
- MLE
- Price_Mom
- EMA_Div
- Entropy_Delta
- Entropy_Curv
- MLE_Delta
- EKF_Res_Delta
- EMA_Div_Delta
- Compression

这意味着真正的模型输入不是单一线性指标，而是**原始特征 + 差分特征 + 结构特征**的组合。

#### 约束三：模型不仅看“当前值”，还看完整序列上下文

`symbolic_extractor.py` 的解释结果明确给出两个结论：

- 模型会吸收完整窗口上下文，而不是只看最后一根
- Pine 落地建议是：
  - **核心物理背景统一用 20 周期**
  - **2~3 周期仅用于短时平滑或锚点确认**

这说明当前体系天然包含：

- 一层“慢背景”
- 一层“快确认”
- 一层“状态切换解释”

#### 约束四：Pine 端已经是压缩后的平台适配表达

现有 `KHAOS.pine` 虽然保持了双核主叙事，但它本质上已经是：

- 对 physics 特征的**在线近似重算**
- 对训练结论的**规则化表达**
- 对注意力锚点的**短窗近似模拟**

因此，当前 Pine 更像是“**研究结果的终端解释器**”，而不是训练模型本体。

### 2.3 已经在文档中暴露出的跨平台问题

现有文档已经反复出现三类问题：

- **特征定义不一致**
  - Python 端的严格物理量，与 Pine 端的代理公式不完全等价
- **归一化与分布漂移**
  - 训练侧与实盘侧使用的统计基准不同，阈值会漂移
- **多周期统一阈值在终端侧容易失真**
  - breakout 在高周期更稀疏，若强行统一表达，会让图表含义失真

这三点决定了：**跨平台方案的核心不是“移植代码”，而是“控制语义损耗”。**

## 3. 为什么 Pine / 同花顺 难以无损承载当前体系

### 3.1 先说结论

Pine 与同花顺这类终端公式环境，适合承载：

- 轻量特征计算
- 规则判断
- 图形展示
- 告警触发

但不适合无损承载：

- 完整训练模型
- 复杂注意力与样条非线性
- 大量版本化参数管理
- 研究级特征对齐与回放验证

### 3.2 Pine 难以无损承载的原因

#### 原因一：表达能力上限不匹配

当前主模型来自 KAN + 双头输出 + 时序上下文，而 Pine 本质上是逐 bar 执行的公式语言。它可以表达近似规则，但不适合承载完整的：

- 高维非线性映射
- 复杂样条基函数
- 训练态的权重体系
- 大规模结构化中间状态

#### 原因二：Pine 端必须使用代理特征

当前 `KHAOS.pine` 中的 Hurst、MLE、Entropy 都是为了在线运行而做过工程压缩的代理实现。只要终端侧不是直接复用训练端同一份特征内核，就一定存在：

- 数值定义偏差
- 阈值解释偏差
- 周期切换偏差

#### 原因三：Pine 更适合“结果消费”，不适合“模型演化”

Pine 指标可以快速展示状态，但不适合承接：

- 模型版本切换
- 参数热更新治理
- 多资产统一服务
- 离线回测与在线一致性核验

也就是说，Pine 是很强的“图表终端”，但不是理想的“模型真值底座”。

### 3.3 同花顺难以无损承载的原因

同花顺公式环境与 Pine 面临的是同类问题，而且通常更强调：

- 终端内公式计算
- 可视化输出
- 平台内调用规范

它同样不适合作为主模型的真值承载层，原因包括：

- 难以直接复用 Python 训练期的完整特征工程
- 难以承载复杂神经网络推理结构
- 难以进行严格版本治理与多终端一致性校准
- 更适合作为“终端侧表达层”，而不是“训练与推理主引擎”

### 3.4 真正的问题不是“能不能写出来”，而是“写出来以后还算不算同一个系统”

很多复杂逻辑在终端公式里并非绝对写不出来，但一旦为了平台限制做以下操作：

- 替换特征定义
- 简化归一化
- 删除上下文依赖
- 压缩模型结构
- 改为硬阈值拼装

那么最终得到的就不是“原 KHAOS 系统的跨平台版本”，而是“**受平台限制重写后的近似版指标**”。

这就是为什么 Pine / 同花顺 难以做到“无损承载”。

## 4. 架构路线对比

| 方案 | 保真度 | 延迟 | 部署复杂度 | 终端适配性 | 适合承担的职责 | 结论 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| Pine | 低到中 | 低 | 低 | 很高 | 图表展示、轻量规则、告警 | 适合作为终端轨，不适合作为真值轨 |
| 同花顺公式 | 低到中 | 低 | 低 | 很高 | 国内终端展示、轻量状态表达 | 适合作为终端轨，不适合作为真值轨 |
| Python | 高 | 中 | 低到中 | 中 | 训练、回测、解释、MVP 推理服务 | 当前最现实的主轨起点 |
| Rust | 高 | 很低 | 中到高 | 中到高 | 高性能推理、统一特征内核、本地库 | 适合作为主轨的后续运行时升级 |
| 本地库 | 高 | 很低 | 中 | 中 | 单机终端、桌面程序集成 | 适合个人/桌面部署，不适合集中治理 |
| 服务端 | 很高 | 中 | 中到高 | 很高 | 多终端统一真值、版本治理、集中告警 | 最适合多平台统一输出 |
| 双轨混合 | 高 | 中 | 中 | 很高 | 主轨保真 + 终端轨分发 | 推荐方案 |

## 5. 推荐架构：双轨混合方案

### 5.1 核心思想

推荐采用：

- **主轨：Python / Rust 真值引擎**
- **副轨：Pine / 同花顺终端表达**

其本质不是“双写两套系统”，而是：

- **只保留一套真值逻辑**
- 针对不同终端生成不同“消费形态”

### 5.2 双轨的直观解释

可以把整个系统理解成“厨房 + 出餐窗口”：

- **主轨是真正做菜的厨房**
  - 负责特征、推理、校准、版本、告警真值
- **副轨是不同平台的出餐窗口**
  - TradingView 一个窗口
  - 同花顺一个窗口
  - 本地桌面软件一个窗口

用户在不同窗口看到的摆盘可以不同，但核心菜品必须来自同一个厨房。

### 5.3 推荐的目标形态

#### 形态 A：短期可落地

- Python 继续承担训练、解释、导出、推理
- 服务端负责对外提供统一结果
- Pine / 同花顺只负责展示状态、读取统一语义

#### 形态 B：中期工程升级

- 保留 Python 作为研究母体
- 新增 Rust 运行时实现统一特征计算与高性能推理
- 本地库与服务端共用同一套 Rust 核心

换言之：

- **Python 负责“研究正确”**
- **Rust 负责“运行高效”**
- **Pine / 同花顺负责“终端可读”**

---

## 6. 职责分层

推荐将系统拆成六层。

### 6.1 数据层

职责：

- 管理多资产、多周期输入
- 统一时间对齐
- 输出标准 OHLCV + 衍生输入

要求：

- 继续保持 5m、15m、1h、4h、1d 五周期完备
- 不允许终端各自定义不同的数据切片方式

### 6.2 特征层

职责：

- 统一生成 Hurst、Volatility、EKF、Entropy、MLE、EMA_Div 及其差分/压缩特征
- 固化 20 周期物理背景窗
- 固化 2~3 周期短确认语义

要求：

- 这一层必须成为**唯一真值特征层**
- Pine / 同花顺如果无法完全复现，只能做近似消费，不得反向定义主标准

### 6.3 推理层

职责：

- 输出 breakout / reversion 双核评分
- 输出 neutral / phase / direction 等标准状态
- 输出告警与风控需要的统一字段

建议输出字段：

- `breakout_score`
- `reversion_score`
- `phase`
- `direction`
- `neutral_filter`
- `anchor_state`
- `model_version`
- `feature_version`

### 6.4 蒸馏与规则层

职责：

- 将真值引擎结果压缩为终端可消费规则
- 输出 Pine / 同花顺可用阈值、映射、状态枚举

这层不是替代真值引擎，而是做：

- 轻量副本
- 终端兼容
- 断网兜底

### 6.5 终端适配层

职责：

- Pine 适配器
- 同花顺适配器
- 本地桌面适配器

只做三件事：

- 接收统一状态
- 绘制统一视觉语义
- 触发终端侧提醒

### 6.6 服务与分发层

职责：

- 提供统一 API / 本地接口
- 下发版本号与参数包
- 记录信号日志
- 做多终端一致性治理

## 7. 性能策略

跨平台系统最怕两件事：

- 每个平台都各算一遍，而且算得还不一样
- 为了“实时”，不断牺牲定义一致性

推荐采用以下性能策略。

### 7.1 先统一特征，再讨论展示

性能优化优先级应该是：

1. 统一特征内核
2. 减少重复计算
3. 终端只消费必要字段

而不是一开始就在 Pine / 同花顺里重复堆复杂公式。

### 7.2 固定慢背景，压缩快确认

现有系统已经给出一个非常好的工程边界：

- **20 周期负责慢背景**
- **2~3 周期负责快确认**

这意味着可以把大量复杂波动留在主轨中，而终端轨只消费：

- 状态结果
- 锚点结果
- 可视化颜色与柱线

### 7.3 增量计算优先

无论主轨最终由 Python 还是 Rust 承担，都应优先采用：

- 滑窗缓存
- 增量更新
- 上下文状态复用

避免每来一根 bar 就整窗全量重算。

### 7.4 本地与服务端共用同一真值协议

如果以后既有本地部署，又有服务端部署，务必做到：

- 同一字段定义
- 同一版本号体系
- 同一阈值解释

否则“本地看多、服务端看空、Pine 看中性”的分裂会迅速出现。

### 7.5 终端只保留必要降级能力

Pine / 同花顺应保留的不是“完整重算能力”，而是：

- 基础状态回显
- 轻量锚点确认
- 无外部连接时的近似兜底

这能让终端跑得稳，也能控制语义漂移。

## 8. 最小落地版本

最小可用版本不建议一开始就上 Rust 全家桶，而是建议分两步。

### 8.1 MVP 版本

#### 架构

- Python 训练与推理核心
- 一个轻量本地或服务端接口
- Pine / 同花顺只做状态展示与轻量规则回显

#### MVP 必须交付的内容

- 一套统一的特征版本定义
- 一套统一的双核输出字段
- 一套可供 Pine / 同花顺 消费的状态映射表
- 一套版本号与参数包导出机制

#### MVP 不必一开始解决的问题

- 不必先把完整推理写成 Rust
- 不必先追求所有终端完全本地离线推理
- 不必先追求零损耗移植

MVP 的目标只有一个：

**先让所有平台“说同一种状态语言”。**

### 8.2 第二阶段版本

- 将统一特征层与推理层抽成 Rust 运行时
- 同时支持：
  - 本地库调用
  - 服务端部署
- Python 保留为训练、验证、解释与导出母体

## 9. 实施顺序

推荐按以下顺序推进。

### 第一步：冻结真值语义

先明确统一输出口径：

- breakout 是什么
- reversion 是什么
- neutral/filter 是什么
- direction 何时有、何时没有
- anchor 只是确认，不是主状态来源

### 第二步：冻结真值特征层

明确哪些特征属于主标准，哪些只是终端代理。

这一步是整个跨平台成功与否的分水岭。

### 第三步：定义统一输出协议

形成标准输出字段与版本规范，供：

- Pine
- 同花顺
- 本地库
- 服务端

共同消费。

### 第四步：先做 Python 主轨 MVP

先让 Python 真值引擎可稳定输出：

- 特征
- 双核分数
- 状态字段
- 参数包

### 第五步：改造 Pine / 同花顺 为消费端

不要再把它们当作“主模型实现地”，而是改造成：

- 可视化解释器
- 告警触发器
- 终端交互层

### 第六步：再决定是否引入 Rust

当出现以下需求时再推进 Rust：

- 高频实时推理
- 大规模多资产并发
- 桌面本地程序集成
- 服务端资源成本优化

## 10. 对 Pine / 同花顺 的最终定位

### 10.1 Pine 的定位

Pine 最适合承担：

- TradingView 图表可视化
- 轻量状态回显
- 告警触发
- 终端交互体验

Pine 不应再被要求承担：

- 完整 PI-KAN 真值推理
- 主特征标准定义
- 多平台一致性的唯一来源

### 10.2 同花顺的定位

同花顺最适合承担：

- 国内终端环境中的状态展示
- 与 Pine 平行的消费端表达
- 面向中文用户的终端侧交互

同花顺也不应被要求承担：

- 研究级训练逻辑
- 主模型真值治理
- 复杂神经网络本体迁移

### 10.3 统一定位总结

Pine / 同花顺都应该被定义为：

**“平台终端层”**

而不是：

**“模型主引擎层”**

---

## 11. 最终建议

如果目标是“跨平台可维护、语义一致、后续还能继续迭代”，那么最合理的路线不是“All in Pine”，也不是“All in 同花顺”，而是：

### 推荐结论

- **短期推荐**
  - 以 Python 作为主轨真值引擎
  - 以 Pine / 同花顺作为副轨终端表达
- **中期推荐**
  - 在 Python 研究母体稳定后，引入 Rust 统一特征与推理运行时
  - 同时支持本地库与服务端两种部署方式
- **长期推荐**
  - 保持“一套真值引擎 + 多终端消费层”的体系，不再让终端公式语言反向定义主系统

### 一句话结论

**Pine / 同花顺 负责“把结果讲清楚”，Python / Rust 负责“把结果算正确”，双轨混合负责“把整个系统做成能长期维护的工程”。**
